SQLite 与 PostgreSQL：选型指南

编码 2025-10-10 · 0 次阅读 · 预计阅读 10 分钟

最近在做一个新项目，需要选择数据库。在 SQLite 和 PostgreSQL 之间做选择时，我花了一些时间仔细对比了两者的异同。这篇文章记录了我的思考过程和选型建议。

先说结论：SQLite 和 PostgreSQL 没有绝对的优劣，只有是否适合你的场景。

对比总览

首先用一个表格来直观对比两个数据库的核心差异：

特性           | SQLite                    | PostgreSQL
--------------+---------------------------+--------------------------
类型          | 嵌入式关系型数据库        | 客户端-服务器关系型数据库
部署          | 零配置，文件即可           | 需要独立服务进程
并发写入      | 互斥锁，一次只有一个写入   | MVCC，并发读写
数据大小      | 单文件，通常建议 < 2TB    | 不限
SQL 标准      | 基础支持，部分缺失         | 高度兼容
扩展性        | 一般                       | 丰富（扩展、插件）
全文搜索      | 内置 FTS5                  | 需配置
JSON 支持     | 有（JSON1 扩展）           | 强大（JSONB）
复制          | 不内置                     | 流复制
平台          | 所有平台                   | 主要平台

SQLite 的优势

零配置

SQLite 不需要安装、配置或管理。你只需要一个数据库文件，就可以开始使用。这极大地降低了上手门槛，也减少了运维成本。

轻量级

SQLite 库本身只有几百 KB，运行时内存占用极低。它甚至可以运行在嵌入式设备上。我曾在 OpenWrt 路由器上使用 SQLite 存储日志数据，运行得非常稳定。

可靠性

SQLite 的开发者对可靠性的要求极高。它拥有业界最严格的测试套件之一，包括 100% 的分支覆盖率测试、内存耗尽测试、磁盘 I/O 错误模拟等等。SQLite 的开发者表示，SQLite 的测试代码量是核心代码量的数百倍。

-- SQLite 全文搜索示例
CREATE VIRTUAL TABLE docs USING fts5(title, content);

INSERT INTO docs VALUES ('SQLite 入门', 'SQLite 是一个轻量级的嵌入式数据库');
INSERT INTO docs VALUES ('PostgreSQL 进阶', 'PostgreSQL 是一个功能强大的关系型数据库');

SELECT * FROM docs WHERE docs MATCH '轻量级';
-- 返回: SQLite 入门 | SQLite 是一个轻量级的嵌入式数据库

PostgreSQL 的优势

功能全面

PostgreSQL 常被称为"开源界的 Oracle"。它拥有极其丰富的功能集：

  • 完整的 SQL 标准支持（包括窗口函数、CTE、递归查询等）
  • 丰富的数据类型（数组、JSONB、范围类型、枚举、UUID 等）
  • 强大的索引机制（B-tree、Hash、GiST、GIN、BRIN 等）
  • 内置的复制和分区支持
  • 存储过程和自定义函数
  • 外部数据包装器（FDW）

并发处理

PostgreSQL 使用 MVCC（多版本并发控制）来处理并发访问。读操作不会被写操作阻塞，写操作也不会被读操作阻塞。这使得 PostgreSQL 非常适合高并发的 Web 应用场景。

-- PostgreSQL JSONB 查询示例
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    data JSONB NOT NULL
);

INSERT INTO events (data) VALUES
    ('{"type": "click", "user": "alice", "page": "/home"}'),
    ('{"type": "view", "user": "bob", "page": "/about"}');

-- GIN 索引加速 JSONB 查询
CREATE INDEX idx_events_data ON events USING GIN (data);

-- 查询特定条件
SELECT * FROM events WHERE data @> '{"type": "click"}';

选型决策建议

选择 SQLite 的场景

  • 嵌入式应用：移动 App、桌面软件、IoT 设备
  • 开发和测试：本地开发环境、单元测试（可以极大简化 CI/CD 配置）
  • 数据分析：数据分析师的本地查询工具
  • 小规模网站：低并发、单服务器、数据量在 GB 级别以内
  • 临时工具：原型验证、内部工具、批处理脚本

选择 PostgreSQL 的场景

  • Web 应用：多用户、高并发、需要实时读写
  • 地理空间数据：PostGIS 扩展提供了强大的地理信息处理能力
  • 复杂查询：需要窗口函数、递归 CTE、物化视图等高级 SQL 功能
  • 数据完整性要求高：外键约束、检查约束、触发器
  • 需要高可用：流复制、主从切换、负载均衡
  • 多应用共享数据：不同应用访问同一数据库，需要精细的权限控制

一个实用的策略

我发现一个非常有效的策略：开发时用 SQLite，生产环境用 PostgreSQL。

这个策略之所以可行，是因为两个数据库都支持标准的 SQL，熟练之后可以写出兼容两者的代码。开发阶段使用 SQLite 可以省去配置数据库服务的麻烦，而生产环境使用 PostgreSQL 可以获得更高的并发能力和丰富的功能。

但需要注意，不是所有 SQL 都能在两个数据库之间无缝迁移。一些数据类型和函数在不同数据库中的表现可能存在差异。建议在切换数据库时进行充分的测试。

总结

SQLite 是"够用就好"哲学的完美体现。它不需要你操心任何运维问题，开箱即用。PostgreSQL 则是"功能完备"的代表，提供了企业级的功能和可靠性。

选择哪一个，取决于你的具体需求。如果可以在两者之间妥协，我的建议是：用 SQLite 做简单的事，用 PostgreSQL 做重要的事。
