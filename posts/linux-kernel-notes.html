Linux 内核学习笔记：进程调度

编码 2025-11-15 · 0 次阅读 · 预计阅读 15 分钟

进程调度是操作系统最核心的功能之一。Linux 内核的调度器经历了数次重写，从最初的 O(n) 调度器，到 O(1) 调度器，再到 2.6.23 版本引入的 CFS（Completely Fair Scheduler，完全公平调度器），每一次演进都朝着更公平、更高效的方向前进。

调度器演进简史

Linux 调度器的发展大致经历了以下几个阶段：

  • O(n) 调度器（Linux 2.4 及之前）：每次调度时遍历所有进程，时间复杂度与进程数成正比
  • O(1) 调度器（Linux 2.6.0-2.6.22）：引入优先级数组，调度时间恒定；但交互式进程判定启发式规则复杂
  • CFS 调度器（Linux 2.6.23 至今）：基于红黑树实现完全公平调度，取代了复杂的优先级数组

CFS 的核心思想

CFS 的设计理念非常朴素：如果系统中有 N 个可运行进程，理想情况下每个进程应该获得 1/N 的 CPU 时间。CFS 使用 "虚拟运行时间"（vruntime）来跟踪每个进程已使用的 CPU 时间，并总是选择 vruntime 最小的进程来运行。

vruntime = 实际运行时间 × (nice_0_weight / 当前进程权重)

通过这个公式，高优先级的进程（权重更大）在相同实际运行时间下，vruntime 增长更慢，从而获得更多的 CPU 时间。

数据结构：红黑树

CFS 使用红黑树来组织可运行进程，以 vruntime 为键值：

struct cfs_rq {
    struct rb_root tasks_timeline;
    struct rb_node *rb_leftmost;  // 缓存最左节点（vruntime 最小的进程）
    struct sched_entity *curr;     // 当前正在运行的进程
    u64 min_vruntime;             // 队列中最小的 vruntime
    unsigned int nr_running;      // 可运行进程数
    ...
};

红黑树在最左叶子节点保存 vruntime 最小的进程。由于 CFS 每次都选择最左节点运行，且红黑树的插入和删除操作都是 O(logN)，调度效率很高。同时，rb_leftmost 缓存了最左节点，使得选择下一个进程的操作是 O(1)。

调度周期与时间片

CFS 不再使用固定的时间片概念，而是引入了"调度周期"（sched_latency）。在一个调度周期内，每个可运行进程至少运行一次。

// 进程获得的时间片
timeslice = 调度周期 × (进程权重 / 所有进程总权重)

// 默认值
sched_latency = 6ms
min_granularity = 0.75ms

当可运行进程数增多时，每个进程的时间片会变小，上下文切换开销会增加。因此当进程数超过一定阈值（sched_nr_latency），CFS 会放大调度周期：

// 当 nr_running > sched_nr_latency 时
sched_period = max(sched_latency, nr_running * min_granularity)

调度类（Scheduling Classes）

从 Linux 2.6.38 开始，CFS 不再是唯一的调度器，而是作为调度类体系的一部分。Linux 内核实现了可插拔的调度类，按优先级从高到低：

  • Stop 调度类：优先级最高，用于停止 CPU
  • Deadline 调度类：使用 EDF（Earliest Deadline First）算法，用于实时任务
  • RT 调度类：实时调度，包括 SCHED_FIFO 和 SCHED_RR
  • CFS 调度类：普通进程调度，包括 SCHED_NORMAL 和 SCHED_BATCH
  • Idle 调度类：空闲任务

调度时，内核按照优先级从高到低遍历各个调度类，每个调度类提供自己的可运行进程选择逻辑。

调度点（Schedule Points）

调度不会随便发生，而是发生在特定的调度点：

  • 进程主动调用 schedule()（如等待 I/O）
  • 时钟中断触发周期性调度（检查当前进程是否用完时间片）
  • 进程被唤醒（如等待的 I/O 完成）
  • 进程创建（fork 后调度父进程还是子进程）
  • 进程改变优先级或策略

其中最重要的是时钟中断处理函数 scheduler_tick()，它更新当前进程的 vruntime，检查是否需要抢占，并在必要时设置 TIF_NEED_RESCHED 标志。

组调度（Group Scheduling）

CFS 的一个强大特性是支持组调度。普通情况下，CFS 调度的是进程。但开启组调度后，CFS 将用户组视为调度实体，组内的进程共享该组获得的 CPU 时间。

这对于多用户系统非常有用：如果用户 A 启动了 10 个进程，用户 B 只启动了 1 个进程，没有组调度时，用户 A 会获得约 91% 的 CPU 时间。开启组调度后，两个用户各获得 50%，然后各自在组内分配。

总结

CFS 的设计哲学是通过简单的数据结构和清晰的算法来达到公平调度的目标。相比于 O(1) 调度器中复杂的启发式规则，CFS 更加可预测和易于理解。这也是 Linux 内核社区追求简洁设计的体现。

当然，CFS 并非完美。在面临大量短生命周期进程或极端实时性要求时，CFS 仍然有其局限性。这也是为什么内核社区一直在持续优化调度器，比如引入 BORE 调度器、EEVDF 等改进方案。
